# KnowledgeMelon 项目学习文档

这份文档面向“已经会做全栈开发，但刚开始学习 AI 工程”的同学。
目标不是教你所有 AI 理论，而是结合 `KnowledgeMelon` 这个项目，帮你快速建立一个能看懂、能改、能扩展的认知框架。

---

## 1. 这个项目到底在做什么

`KnowledgeMelon` 是一个本地优先的 AI 工作台。它把四件事放在一起：

1. 聊天
2. 知识库
3. 记忆
4. 工具调用

你可以把它理解成一个“带长期上下文能力的 AI 办公桌”。

普通聊天应用只处理当前一句话。
这个项目的重点是让模型可以：

- 记住历史对话
- 读取你上传的文档
- 记住你长期偏好
- 在需要时调用工具，比如联网搜索

---

## 2. 项目架构总览

这个仓库是典型的前后端分离结构：

- `frontend/`：Next.js 前端
- `backend/`：FastAPI 后端
- `docker-compose.yml`：本地 PostgreSQL + pgvector

### 前端负责什么

前端主要负责：

- 页面展示
- 状态切换
- 文件上传
- 主题/语言切换
- 聊天流式输出展示
- 知识库和记忆管理 UI

### 后端负责什么

后端主要负责：

- 聊天请求转发和编排
- 文档解析、切块、向量化
- 知识库检索
- 记忆提取与记忆检索
- 模型调用
- 工具调用

你可以把前端看作“操作台”，把后端看作“AI 编排器”。

## 3. 你需要先建立的 AI 基础概念

### 3.1 大模型不是数据库

模型本身不会天然记住你的文档、记忆或历史对话。
它能“知道”的内容，通常来自三种方式：

- 直接在 prompt 里放进去
- 通过检索系统找到相关内容再放进去
- 通过工具调用临时获取信息

这个项目就是把这三种能力组合起来。

### 3.2 Prompt

Prompt 可以理解为发给模型的说明书。
在这个项目里，最终发给模型的输入通常不是单句用户消息，而是：

- system prompt
- 历史消息
- 知识库检索结果
- 记忆内容
- 当前会话附件
- 用户当前问题

### 3.3 RAG

RAG = Retrieval Augmented Generation，检索增强生成。

简单说就是：

1. 先去知识库里找相关内容
2. 再把找来的内容交给模型
3. 模型基于这些内容回答

它的意义是让模型回答你的私有资料，而不是只靠训练时学到的通用知识。

### 3.4 Memory

Memory 不是“长聊天记录”。
更准确地说，它是被筛选出来的长期可用信息，例如：

- 你喜欢的表达方式
- 你的项目目标
- 你的身份信息
- 你不希望模型重复问的问题

记忆通常是结构化的、可检索的、可长期保存的。

### 3.5 Tool Calling

工具调用是让模型不只会“说”，还能“做”。

比如：

- 联网搜索
- 查天气
- 调外部 API
- 执行用户自定义工具

模型会先判断要不要调用工具，后端负责真正执行，再把结果交回模型。

## 4. 这个项目里的核心数据流

这是最重要的一部分。

### 4.1 普通聊天流

当你在 `/chat` 输入消息时，大致流程是：

1. 前端收集输入、模型选择、开关状态
2. 前端调用后端 `/api/chat` 或 `/api/chat/rag`
3. 后端构造上下文
4. 后端调用大模型
5. 大模型流式输出
6. 前端边接收边渲染

### 4.2 同时打开知识库、记忆、联网搜索时

如果三个开关都开了，后端会把这些信息合并到一次请求里：

- 知识库检索结果
- 记忆上下文
- 工具调用能力

最终由模型自己决定：

- 直接回答
- 优先参考知识库
- 需要最新信息时调用联网搜索

注意：

- 知识库和记忆是“上下文输入”
- 联网搜索是“可调用工具”

它们不是同一种机制。

## 5. 前端页面该怎么理解

### 5.1 `/chat`

这是最核心的页面。

你可以把它理解成：

- 左侧：会话导航
- 中间：消息展示
- 底部：输入与附件
- 顶部：模型选择与功能开关

这里你会看到：

- 流式回答
- 文件附件
- 语音输入
- RAG / 记忆 / 联网搜索开关

### 5.2 `/knowledge`

知识库页负责：

- 上传文档
- 查看解析结果
- 处理向量化进度
- 删除文档

它的本质是“给 RAG 提供材料”。

### 5.3 `/memories`

记忆页负责：

- 查看记忆
- 新增/删除记忆
- 配置自动提取规则

它的本质是“给模型提供长期偏好和事实”。

### 5.4 `/settings`

设置页负责：

- 模型选择
- API Key 配置
- 主题切换
- 语言切换

这是项目的“运行环境控制面板”。

## 6. 后端模块怎么分工

### `backend/api/`

这里是路由层。
它负责接收 HTTP 请求，把请求交给 service 层。

你可以把它理解成 Controller。

### `backend/services/`

这里是业务层。
常见职责包括：

- 解析文档
- 生成 embeddings
- 做向量检索
- 组织聊天上下文
- 调用模型
- 处理记忆

### `backend/models/`

这里是数据结构层：

- SQLAlchemy 表
- Pydantic 请求响应模型

### `backend/core/`

这里通常放基础设施：

- 数据库连接
- 配置读取
- 启动时初始化

## 7. 知识库是怎么工作的

知识库这条链路可以拆成 4 步：

1. 上传文件
2. 解析文件文本
3. 切块
4. 生成向量并入库

### 7.1 为什么要切块

大模型不适合直接吞整本长文档。
所以会先把文档切成较小文本块。

每个块会单独生成 embedding。

### 7.2 什么是 embedding

embedding 可以理解成“语义向量”。

它把文本变成一个数字数组，让系统能算出：

- 哪些文本意思相近
- 哪些文本和你的问题相关

### 7.3 pgvector 的作用

pgvector 让 PostgreSQL 可以存向量、做向量相似度搜索。

这意味着：

- 文档文本存在数据库或文件系统
- 文档向量存在 PostgreSQL 的向量字段里
- 查询时根据语义相似度找相关块

---

## 8. 记忆是怎么工作的

记忆模块的目标不是存“聊天记录备份”，而是存“可复用的信息”。

常见记忆类型：

- fact：事实
- preference：偏好
- goal：目标
- important：重要事项

工作方式通常是：

1. 从对话中提取潜在记忆
2. 存到数据库
3. 后续聊天时按相关性检索
4. 把相关记忆放到模型上下文里

这和知识库很像，但语义上不同：

- 知识库是“外部资料”
- 记忆是“关于用户本身的信息”

---

## 9. 联网搜索是怎么工作的

联网搜索是工具调用的一种。

它和 RAG 的区别：

- RAG 搜索的是你自己的资料
- 联网搜索搜索的是外部世界

在这项目里，联网搜索不是默认无脑联网，而是：

1. 你开启“联网搜索”开关
2. 模型在需要时决定是否调用 `web_search`
3. 后端执行搜索
4. 搜索结果回传给模型

这样做的好处是：

- 更可控
- 成本更低
- 不会每次都去访问外部网络

---

## 10. 当前会话附件是什么

你现在在聊天页上传文件时，默认会先作为“当前会话附件”生效。

这和知识库入库是两条不同路径：

- 当前会话附件：只影响这一次对话
- 加入知识库：长期可检索复用

可以把它理解成：

- 附件 = 临时上下文
- 知识库 = 长期资产

这非常适合知识型聊天：

- 临时发个文件给模型看
- 但不一定要把它永久存入知识库

## 11. 这个项目的 AI 请求优先级

如果你打开了多个能力，常见优先级可以理解成：

1. 当前会话历史
2. 当前会话附件
3. 知识库相关内容
4. 记忆相关内容
5. 工具调用结果

注意这不是死板的绝对规则，而是这个项目实际组织上下文的方式。

你可以把它理解成：

- 历史决定对话连续性
- 附件补充当前任务材料
- 知识库补充私有资料
- 记忆补充用户画像
- 工具补充外部实时信息

## 12. 学这个项目时最容易混淆的点

### 12.1 知识库和记忆的区别

知识库是资料库。

记忆是用户画像和长期偏好库。

### 12.2 RAG 和联网搜索的区别

RAG 搜索的是你自己的文档。

联网搜索搜索的是外部互联网。

### 12.3 模型和工具的区别

模型负责“理解和生成”。

工具负责“获取信息或执行动作”。

### 12.4 历史消息和记忆的区别

历史消息是对话过程。

记忆是从对话里提炼出来、可以长期保留的事实。

---

## 13. 如果你要扩展这个项目，推荐的学习顺序

如果你是全栈工程师，但刚接触 AI，我建议按这个顺序学：

1. 先理解聊天请求是怎么流式返回的
2. 再看知识库上传和向量检索
3. 再看记忆提取和记忆检索
4. 再看工具调用
5. 最后再看如何接入更复杂的 agent 工作流

这样不会一上来就被“AI agent / RAG / memory / tool calling”这些词绕晕。
