# KnowledgeMelon 项目学习文档 - 代码导读版

这份文档适合一边看代码一边学习。

目标很简单：

- 你知道每个文件负责什么
- 你知道数据怎么从前端流到后端
- 你知道 AI 能力是怎么被拼起来的
- 你不会被“RAG / Memory / Tool Calling / Embedding”这些词绕晕

---

## 1. 先看仓库主线

这个项目可以分成三层：

1. 前端页面层
2. 后端业务层
3. AI 能力编排层

你可以把它想成：

- 前端负责“让用户操作”
- 后端负责“接住请求并组装上下文”
- AI 层负责“把大模型、知识库、记忆和工具连起来”

---

## 2. 前端导读

### 2.1 `frontend/app/layout.tsx`

这是全站布局入口。

你会在这里看到：

- `<html>` 和 `<body>` 的基础结构
- 主题 Provider
- 语言 Provider
- 全局 metadata

它的作用是：

- 给整个前端提供统一外壳
- 决定页面加载时的全局环境

你可以先关注这里的三个概念：

- `LocaleProvider`
- `ThemeProvider`
- `metadata`

---

### 2.2 `frontend/components/app-shell.tsx`

这是全站侧边栏壳子。

负责：

- 左侧导航
- 应用名展示
- 页面之间统一结构

导航项包括：

- 首页
- 对话
- 知识库
- 记忆
- 设置

如果你想理解“整个应用的页面框架”，先看这个文件。

---

### 2.3 `frontend/app/page.tsx`

首页。

它的作用主要是：

- 介绍产品
- 引导进入聊天、知识库、记忆

这里没有复杂的 AI 逻辑，主要是入口和导览。

---

### 2.4 `frontend/app/chat/page.tsx`

这是最重要的前端文件。

你在这里会看到：

- 会话列表
- 模型选择
- RAG / Memory / Web Search 开关
- 文件上传
- 语音输入
- 消息流式渲染
- 删除会话
- 重新生成回答

你可以把这个页面拆成 5 块看：

1. 左侧栏
2. 顶部模型区域
3. 中间消息区域
4. 底部输入区域
5. 知识库附件和弹窗

#### 重点看什么

- `handleSend`
  - 这是发送消息的入口
  - 决定调用 `/api/chat` 还是 `/api/chat/rag`
- `sessionAttachments`
  - 当前会话附件
- `pendingKnowledgeAttachments`
  - 弹窗里待选择是否入库的文件
- `useRAG / useMemory / useTools`
  - 三个 AI 功能开关

#### 你要理解的关键点

这个页面不是“直接把文字发给模型”。

它会先把这些东西整理好：

- 当前输入
- 历史消息
- 附件内容
- 功能开关
- 模型信息

然后再发给后端。

---

### 2.5 `frontend/app/knowledge/page.tsx`

知识库页。

负责：

- 上传文档
- 展示文档列表
- 查看解析内容
- 删除文档
- 显示处理进度

这里重点看：

- `processFile`
  - 上传文档的主入口
- `handleViewDocument`
  - 打开文档预览
- `loadMorePages`
  - 大文档分页加载

#### 为什么这里重要

因为知识库不是“上传完就结束”。

真正关键的是：

1. 文档解析
2. 文本切块
3. 向量生成
4. 后续检索

这个页面展示的是第一步到第四步的结果。

---

### 2.6 `frontend/app/memories/page.tsx`

记忆页。

负责：

- 展示记忆列表
- 手动新增记忆
- 删除记忆
- 设置自动提取规则

你可以把它理解成“用户长期信息管理页”。

重点看：

- `handleAddMemory`
- `handleDelete`
- `handleSaveSettings`

---

### 2.7 `frontend/app/settings/page.tsx`

设置页。

负责：

- 模型选择
- API Key 配置
- 主题切换
- 语言切换

如果你想理解“这个项目怎么支持多模型、多语言”，看这里最直接。

重点看：

- `getLocalizedModels(locale)`
- `setLocale`
- `setTheme`
- `setModel`

---

### 2.8 `frontend/stores/settings.ts`

这是前端状态核心文件。

它负责存：

- 当前模型
- 主题
- 语言
- API Key
- Base URL
- 功能开关

你可以把它理解成前端版的“配置中心”。

这里最重要的几个函数：

- `getEffectiveApiKey()`
- `setUseRAG()`
- `setUseMemory()`
- `setUseTools()`
- `setUseLocalEmbedding()`

#### 重点理解

这个 store 决定了：

- 当前聊天到底用哪个模型
- 用不用知识库
- 用不用记忆
- 用不用联网搜索
- 知识库上传时用本地 embedding 还是云 embedding

---

### 2.9 `frontend/lib/i18n.ts`

国际化基础。

负责：

- 当前语言读取
- 文案翻译
- `t(key, values)` 的字符串替换

你可以把它理解成：

- 页面里所有显示文字，尽量都从这里拿
- zh / en 的切换都从这里控制

---

### 2.10 `frontend/lib/api.ts`

API 地址入口。

通常你会在这里看到：

- `API_BASE_URL`

它决定前端请求后端的地址。

## 3. 后端导读

### 3.1 `backend/main.py`

后端程序入口。

负责：

- 初始化 FastAPI
- 注册路由
- 启动时初始化数据库
- 配置 CORS

它相当于整个后端的“总开关”。

---

### 3.2 `backend/api/chat.py`

聊天 API 入口。

这是 AI 编排最核心的路由之一。

这里会处理：

- 普通聊天 `/api/chat`
- 带 RAG / memory / tools 的聊天 `/api/chat/rag`
- 临时附件解析 `/api/chat/attachments/preview`

#### 你要理解的关键点

这个文件负责“把前端的开关变成真正的 AI 上下文”。

它会把：

- 历史消息
- 知识库上下文
- 记忆上下文
- 当前会话附件
- 工具开关

传给 `llm_service`。

---

### 3.3 `backend/api/documents.py`

知识库文档 API。

负责：

- 上传文档
- 列出文档
- 获取文档内容
- 获取 PDF 预览信息
- 获取原文件
- 删除文档
- 搜索文档

#### 最重要的两条链路

1. 上传链路
   - 文件上传
   - 保存文件
   - 解析内容
   - 生成 chunk
   - 生成 embedding
   - 入库

2. 预览链路
   - 根据文档 ID 获取内容
   - markdown 文档直接渲染为文本
   - PDF 做分页预览

---

### 3.4 `backend/api/memories.py`

记忆 API。

负责：

- 读取记忆
- 添加记忆
- 删除记忆
- 配置记忆规则

你可以把它理解成“用户长期信息的 CRUD 层”。

---

### 3.5 `backend/api/conversations.py`

会话 API。

负责：

- 创建会话
- 列出会话
- 搜索会话
- 获取会话详情
- 删除会话

它的角色是：

- 让聊天可以“有标题、有历史、有列表”
- 不只是一个临时的单轮问答框

---

### 3.6 `backend/services/llm_service.py`

这是整个 AI 编排的核心。

负责把所有上下文拼成模型输入：

- 历史消息
- RAG 检索结果
- 记忆内容
- 当前附件内容
- 工具调用

然后调用模型流式输出。

#### 这里要重点看什么

- `_build_messages()`
  - 构造 system / history / user 消息
- `stream_chat()`
  - 真正发起模型调用并流式返回结果
- `_get_web_search_tool_defs()`
  - 决定联网搜索工具是否可用

#### 你要建立的认知

这个文件不是“模型 SDK 包装器”，而是“上下文编排器”。

---

### 3.7 `backend/services/rag_service.py`

RAG 检索服务。

它负责：

- 计算相似文本
- 从向量库里找最相关的 chunk

你可以理解为：

- 输入问题
- 找出知识库中最相近的片段
- 返回给 `llm_service`

---

### 3.8 `backend/services/memory_service.py`

记忆服务。

负责：

- 从对话中提取记忆
- 检索相关记忆
- 为聊天上下文提供记忆内容

---

### 3.9 `backend/services/document_service.py`

文档解析服务。

负责：

- 保存上传文件
- 解析 PDF / DOCX / TXT / MD
- 获取 PDF 页数和大小
- 文本切块辅助

如果你想知道“为什么知识库页能看 markdown、能看 PDF”，先看这个文件。

---

### 3.10 `backend/services/embedding_service.py`

向量生成服务。

负责把文本转成 embedding。

它会根据配置选择：

- OpenAI
- 阿里云
- 智谱
- Moonshot
- 本地 Ollama embedding

这个文件是知识库能做语义检索的关键。

---

### 3.11 `backend/services/tools_service.py`

工具注册和执行服务。

现在项目里最重要的内置工具是：

- `web_search`

后面如果你做自定义工具，这里会继续扩展。

---

## 4. AI 流程导读

这一部分最重要。

你可以把整个项目的 AI 流程理解为四条主线：

1. 聊天主线
2. 知识库主线
3. 记忆主线
4. 工具主线

---

### 4.1 聊天主线

用户在聊天框输入问题。

流程大致是：

1. 前端收集输入
2. 读取当前模型和开关
3. 打包历史消息
4. 传给后端
5. 后端调用 `llm_service`
6. 模型流式返回
7. 前端逐段显示

#### 如果没开任何高级功能

就会走最基础的聊天路径。

#### 如果开了知识库 / 记忆 / 联网搜索

就会走增强路径。

---

### 4.2 知识库主线

流程大致是：

1. 上传文档
2. 保存原文件
3. 解析文本
4. 切块
5. 生成 embedding
6. 存入数据库
7. 聊天时做相关检索

#### 你需要记住

知识库不是“文件浏览器”。

它的真正作用是：

- 给模型提供可检索上下文

---

### 4.3 记忆主线

流程大致是：

1. 聊天过程中提取记忆
2. 保存到记忆库
3. 下次聊天时检索相关记忆
4. 作为上下文喂给模型

#### 记忆和知识库的区别

- 知识库来自文档
- 记忆来自用户行为和对话

---

### 4.4 工具主线

当你打开联网搜索时：

1. 模型得到可用工具定义
2. 模型决定是否调用工具
3. 后端执行工具
4. 工具结果回到模型
5. 模型继续回答

#### 这和普通函数调用不一样

不是前端手动去调搜索接口。

而是模型自己决定“我要不要用工具”。

---

## 5. 一次完整请求怎么走

假设你在聊天页输入一句话，并且同时打开：

- 知识库
- 记忆
- 联网搜索

流程会是：

1. 前端收集输入、历史、附件、开关
2. 请求后端 `/api/chat/rag`
3. 后端先拿会话上下文
4. `llm_service` 拼接知识库上下文
5. 拼接记忆上下文
6. 拼接当前附件上下文
7. 如果启用了 tools，就把 web search 暴露给模型
8. 模型开始推理
9. 需要时调用搜索工具
10. 后端返回工具结果
11. 模型继续生成最终回答
12. 前端流式显示

你可以把它理解成：

- 数据先汇总
- 再交给模型
- 模型在上下文里完成推理

## 6. 学这套代码时最值得关注的几个问题

### 6.1 为什么不是直接把全文给模型

因为：

- 太长
- 太贵
- 太慢
- 容易超上下文窗口

所以要切块、检索、再拼上下文。

### 6.2 为什么要把记忆单独分出来

因为记忆和文档不是一类东西。

文档是知识资产。

记忆是用户长期信息资产。

### 6.3 为什么工具要单独做

因为模型本身不会上网、不会查实时数据、不会执行外部请求。

工具调用让它能做这些事情。

### 6.4 为什么聊天要做流式输出

因为用户体验更好：

- 先看到回答在生成
- 不用等完整结束才看到结果

## 7. 推荐你的阅读顺序

如果你是第一次读这套项目，我建议按这个顺序：

1. `frontend/app/chat/page.tsx`
2. `backend/api/chat.py`
3. `backend/services/llm_service.py`
4. `frontend/app/knowledge/page.tsx`
5. `backend/api/documents.py`
6. `backend/services/document_service.py`
7. `backend/services/embedding_service.py`
8. `frontend/app/memories/page.tsx`
9. `backend/api/memories.py`
10. `backend/services/memory_service.py`

这样你会先建立“聊天主线”，再补知识库、记忆、工具。

---

## 8. 一句话总结

这套项目的核心不是“写了个聊天页面”，而是：

> 用前端收集用户意图，用后端编排知识库、记忆和工具，用大模型统一完成回答。

如果你把这条主线读懂了，你就已经掌握了大部分 AI 应用开发的核心思路。
