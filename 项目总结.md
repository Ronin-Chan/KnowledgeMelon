# 项目总结

这份文档用于系统梳理 KnowledgeMelon 的整体架构、各模块实现方式，以及它为什么会这样设计。你可以把它当成一份“读代码路线图”，按下面顺序学习会比较顺。

---

## 1. 项目定位

KnowledgeMelon 不是一个单纯的聊天页面，而是一个带知识库、记忆、对话管理、指标面板的 RAG 工作台。

它的核心目标有三个：

1. 让用户上传自己的知识文件，并通过检索增强回答。
2. 让系统具备长期记忆能力，能记住用户偏好和重要事实。
3. 让整个问答过程可管理、可观察、可回顾。

从功能上看，它已经包含：

- 登录和用户隔离
- 聊天对话
- 知识库上传与检索
- 长期记忆管理
- 对话摘要
- 指标汇总面板
- 多模型和多 provider 配置

---

## 2. 整体架构

项目采用的是典型的前后端分离结构：

- 前端：Next.js App Router
- 后端：FastAPI + SQLAlchemy async
- 数据库：PostgreSQL + pgvector
- 模型接入：通过统一的 LLM/Embedding 服务封装

你可以把整个系统理解成四层：

1. **交互层**
   - 聊天页
   - 知识页
   - 记忆页
   - 设置页
   - 指标页

2. **API 层**
   - `/api/chat`
   - `/api/chat/rag`
   - `/api/documents`
   - `/api/memories`
   - `/api/conversations`
   - `/api/metrics`
   - `/api/auth`
   - `/api/ollama`

3. **业务层**
   - conversation service
   - document service
   - rag service
   - memory service
   - metrics service
   - llm service
   - embedding service

4. **数据层**
   - users
   - documents
   - document_chunks
   - conversations
   - messages
   - memories
   - memory_settings
   - interaction_metrics

## 3. 前端是怎么组织的

前端使用 Next.js 的 App Router，每个功能页基本都是一个独立页面。

### 3.1 全局壳层

`frontend/components/app-shell.tsx` 是整个前端的导航壳层。

它负责：

- 左侧菜单
- 移动端抽屉导航
- 登录态判断
- 语言切换
- 用户信息展示

菜单结构大致是：

- Home
- Chat
- Knowledge
- Memories
- Settings
- Insights

也就是说，前端不是散页，而是统一挂在一个壳里，体验更像一个完整应用。

### 3.2 设置状态

`frontend/stores/settings.ts` 用 Zustand 管理全局模型设置。

这里保存的是用户最常用的运行参数，比如：

- 当前模型
- provider
- API Key
- base URL
- 是否启用 RAG
- 是否启用 Memory
- 是否启用 Tools
- 是否启用本地 embedding
- 响应长度
- 语言
- 主题

这类设计的好处是：

- 页面之间共享配置很方便
- 用户切换模型时不会丢状态
- 设置可以持久化

### 3.3 聊天页面

`frontend/app/chat/page.tsx` 是交互最复杂的页面。

它支持：

- 新建对话
- 历史对话搜索
- 发送消息
- 流式接收回答
- 附件上传
- 语音输入
- RAG 引用展示
- 对话删除
- 重新生成回答

这个页面的核心工作不是“展示一段文本”，而是把聊天、附件、知识检索、记忆、语音、历史对话串成一个完整工作流。

### 3.4 知识页

`frontend/app/knowledge/page.tsx` 负责文档上传、文档列表、文档状态和内容查看。

它现在已经可以展示：

- 文档版本
- 是否最新
- 文档状态
- 文件来源名称
- 文档内容预览

### 3.5 记忆页

`frontend/app/memories/page.tsx` 用来管理长期记忆。

用户可以在这里查看：

- 记忆内容
- 记忆类别
- 重要性
- 来源
- 过期策略相关设置

### 3.6 指标页

`frontend/app/insights/page.tsx` 是项目的汇总面板。

它会展示：

- 使用量
- 检索质量
- 回答质量
- 知识库状态
- 记忆状态

这个页面相当于把后端采集的数据做了一个前端仪表盘。

## 4. 后端是怎么分层的

后端以 FastAPI 为入口，`backend/main.py` 把各个 router 挂到应用上。

启动时会做两件事：

- 初始化数据库
- 注册所有 API 路由

当前包含的核心 router 有：

- chat
- documents
- metrics
- rag
- ollama
- memories
- conversations
- auth

这种结构的优点是：

- 路由职责清晰
- 业务逻辑不会全部堆在一个文件里
- 后续增加模块比较方便

## 5. 数据模型是怎么设计的

`backend/models/entities.py` 是整套系统最重要的基础之一。

### 5.1 User

用户表，保存：

- 用户名
- 邮箱
- 密码哈希
- 创建时间

它还关联了：

- documents
- conversations
- memories
- memory_settings
- api_keys
- interaction_metrics

### 5.2 UserApiKey

用户 API Key 表。

它的作用是：

- 按 provider 存储加密后的 API Key
- 存储 base URL
- 同一个用户同一个 provider 只保留一条配置

### 5.3 Document 与 DocumentChunk

Document 是文档主表，DocumentChunk 是切块表。

Document 里存：

- 文件标题
- 文件类型
- 文件路径
- 内容 hash
- 版本号
- 是否最新
- 来源名称
- 元数据 JSON
- 状态

DocumentChunk 里存：

- chunk 内容
- chunk 序号
- chunk hash
- token 数
- embedding

这套设计说明知识库不是“只存原文件”，而是“原文件 + 切块 + 向量”三层结构。

### 5.4 Conversation 与 Message

Conversation 是会话主表，Message 是对话消息表。

Conversation 里有：

- 标题
- 模型
- 消息数
- token 总量
- summary
- summary_tokens
- 是否激活

Message 里有：

- role
- content
- tokens
- 是否已摘要

这表示系统不只是保存聊天记录，还会做上下文压缩。

### 5.5 Memory 与 MemorySetting

Memory 用来存长期记忆。

它包含：

- 内容
- 类别
- 重要性
- 来源
- normalized_key
- 状态
- expires_at
- confidence
- metadata_json
- embedding
- access_count
- last_accessed

MemorySetting 用来保存记忆相关配置。

### 5.6 InteractionMetric

这是指标系统的核心表。

它记录每次交互的：

- 问题
- 回答
- 模型
- 是否用 RAG
- 是否用 Memory
- 是否用 Tools
- 是否重试
- top_k
- 召回数
- 无结果
- 引用数
- 引用覆盖率
- 检索命中率
- 回答成功
- 是否一次答对
- 是否需要追问
- 幻觉标记
- 人工纠正标记
- 召回来源 JSON

这张表相当于所有指标分析的原始事实表。

## 6. 聊天链路是怎么跑的

聊天接口在 `backend/api/chat.py`。

它有两个主要入口：

- `/api/chat`
- `/api/chat/rag`

### 6.1 普通聊天

普通聊天的流程大致是：

1. 前端发起请求
2. 后端检查登录用户
3. 如果有会话 ID，就先取历史上下文
4. 把用户消息写入 conversation/messages
5. 调用 `llm_service.stream_chat(...)`
6. 边生成边把内容流式返回给前端
7. 回答结束后再把 assistant 消息写库
8. 生成标题
9. 必要时触发摘要
10. 写入指标记录

### 6.2 RAG 聊天

RAG 聊天是在普通聊天基础上增加了“先检索再回答”。

大致流程是：

1. 先根据用户问题去知识库检索相关 chunk
2. 把召回结果作为上下文传给模型
3. 模型生成答案
4. 前端展示答案和引用来源
5. 后端记录这次交互的检索和回答指标

### 6.3 附件处理

聊天页还支持把临时文件作为附件。

支持的类型包括：

- pdf
- docx
- txt
- md

文件会先保存到临时目录，再解析成文本，最后截断到合理长度送入模型。

### 6.4 语音输入

前端还做了浏览器语音识别接入。

这部分主要是提升输入体验，不影响后端主逻辑。

## 7. 知识库是怎么实现的

知识库的实现分为上传、解析、切块、向量化、检索、版本管理几个步骤。

### 7.1 上传与校验

在 `backend/api/documents.py` 里，上传文档时会先检查：

- 文件类型是否支持
- 是否需要 API Key
- 当前用户是否登录

支持的类型目前是：

- pdf
- docx
- txt
- md

### 7.2 内容 hash 去重

上传后会先计算整个文件内容的 hash。

如果同一个用户上传了相同内容，系统会直接识别为重复文档，不会重复入库。

这一步的意义是：

- 降低重复存储
- 避免重复 embedding
- 避免召回时出现大量冗余 chunk

### 7.3 版本控制

如果同一个用户上传的是同名文件的新版本，系统会：

- 计算新的 version
- 把旧版本标记为非最新
- 如果旧版本已经完成，还会把它归档

这样一来，知识库就不会只有“一个文件名对应多个混乱记录”，而是能清楚看到版本关系。

### 7.4 文档解析

文件上传后会进入异步处理流程。

处理步骤是：

1. 保存文件
2. 解析文本
3. 切 chunk
4. 给每个 chunk 做 embedding
5. 写入 `DocumentChunk`
6. 更新文档状态为 completed

### 7.5 Chunk 设计

每个 chunk 额外保存了：

- `chunk_hash`
- `token_count`

这样做的好处是：

- 后面可以做 chunk 去重
- 可以更好地估算上下文长度
- 可以分析 chunk 粒度是否合理

### 7.6 元数据

文档还会保存 `metadata_json`，里面可以继续放：

- 上传者
- 上传时间
- 版本
- 来源
- 额外扩展字段

这为后续做文档治理、筛选和展示提供了基础。

## 8. 检索是怎么做的

检索逻辑在 `backend/services/rag_service.py`。

### 8.1 向量检索

最核心的逻辑是：

1. 把用户问题转成 embedding
2. 去 `DocumentChunk.embedding` 里做 cosine distance 排序
3. 取 top-k chunk
4. 返回给上层作为上下文

### 8.2 用户隔离

检索时会通过 `Document.user_id == user_id` 限制范围。

也就是说：

- 用户只能搜自己的知识库
- 不会跨用户召回别人的文档 chunk

### 8.3 召回预览

除了正式检索，还有一个 `preview_similar` 逻辑。

它会返回：

- 文档标题
- chunk 内容
- score
- reason
- chunk hash

这个设计对前端展示引用来源非常有用。

### 8.4 相关性打分

系统不只是纯向量距离，还会结合词面重叠做一个轻量 score。

也就是说，召回结果里会同时考虑：

- 语义相似
- 关键词匹配

这样比单纯只看向量更稳一点。

## 9. 记忆系统是怎么做的

记忆系统在 `backend/services/memory_service.py`。

### 9.1 记忆的目标

记忆系统不是为了把所有历史都塞进上下文，而是为了保存“值得长期保留的信息”。

比如：

- 用户偏好
- 长期目标
- 稳定事实
- 常用术语

### 9.2 记忆去重

每条记忆都会先生成一个 `normalized_key`。

它是根据：

- category
- content 的标准化文本

计算出的 hash。

这样就可以在写入时判断：

- 这条记忆是不是已经存在
- 是否需要更新而不是新增

### 9.3 TTL 与过期

记忆支持过期时间 `expires_at`。

系统会根据 `memory_ttl_days` 自动计算过期时间。

后台在读写记忆前，还会先清理过期记忆，把状态标成 expired。

### 9.4 冲突策略

记忆设置里支持冲突策略。

当前实现里主要有两种思路：

- `latest_wins`
- `importance_wins`

意思是：

- 要么最新内容覆盖旧内容
- 要么重要性更高的内容优先

这部分是记忆系统能否长期稳定的关键。

### 9.5 自动抽取

系统可以从会话里自动抽取长期记忆。

流程是：

1. 取最近若干轮对话
2. 拼成文本
3. 用 LLM 提取 JSON 格式的事实/偏好/目标
4. 按重要性阈值、白名单、黑名单过滤
5. 写入记忆库

### 9.6 记忆检索

和文档检索类似，记忆也会做 embedding 检索。

查询时会：

- 先清理过期记忆
- 再按向量相似度排序
- 取出最相关的几条
- 增加访问次数与最近访问时间

这说明记忆不仅是“存储层”，也会参与回答生成。

## 10. 对话摘要是怎么做的

对话摘要在 `backend/services/conversation_service.py`。

### 10.1 为什么需要摘要

因为聊天会越聊越长，直接把全部历史塞给模型会：

- 超上下文
- 成本越来越高
- 重要信息被淹没

所以系统采用“摘要 + 最近消息”的方式压缩上下文。

### 10.2 会话数据

Conversation 里保存了：

- 消息数
- token 总量
- summary
- summary_tokens

### 10.3 何时生成摘要

`should_generate_summary` 会根据两个条件判断：

- 消息数达到一定阈值
- 或 token 总量超过阈值

这意味着摘要不是每轮都生成，而是在对话变长后触发。

### 10.4 生成方式

生成摘要时：

1. 取会话里较早的消息
2. 拼成一个摘要 prompt
3. 调用模型生成 summary
4. 把摘要存到 conversation.summary
5. 把已摘要的消息标记为 summarized

### 10.5 上下文组装

发送给模型前，会把：

- summary
- 最近未摘要消息

组合成优化后的上下文。

这样可以兼顾：

- 长期记忆
- 最近对话细节

## 11. 指标系统是怎么做的

指标系统主要在 `backend/services/metrics_service.py` 和 `frontend/app/insights/page.tsx`。

### 11.1 原始记录

每次问答结束后，会写入一条 `InteractionMetric`。

这里记录了：

- 问题
- 回答
- 是否使用 RAG
- 是否使用 Memory
- top_k
- 命中数量
- 无结果
- 引用数
- 引用覆盖率
- 检索命中率
- 回答成功率相关标记
- 幻觉标记
- 人工纠正标记
- 召回来源 JSON

### 11.2 汇总逻辑

`build_summary` 会把原始记录聚合成前端能直接展示的指标。

主要分成四块：

1. 使用量
2. 检索质量
3. 回答质量
4. 知识与记忆状态

### 11.3 前端仪表盘

`insights` 页面会把这些数据展示成：

- 数值卡片
- 进度条
- 最近交互列表

这样你一眼就能看到系统当前运行情况。

## 12. 认证与数据隔离

虽然这是单用户导向的项目，但代码仍然保留了用户级隔离。

主要表现为：

- 大部分查询都按 `user_id` 过滤
- 文档、对话、记忆、指标都只查当前用户
- 登录态通过 `get_current_user` 统一控制

这样做的意义是：

- 结构上仍然是多用户安全模型
- 即使当前只给一个人用，也不会破坏数据边界

## 13. 模型与 provider 是怎么配置的

模型配置主要在 `frontend/stores/settings.ts`。

这里预置了很多模型：

- OpenAI
- Anthropic
- Google
- DeepSeek
- 阿里云
- 智谱
- Kimi
- Mistral

配置思路是：

- 前端选择模型
- 前端选择 provider
- 前端保存 API Key / base URL
- 后端统一通过 LLM 服务执行

这样做的好处是：

- 不需要为每家模型单独写一套聊天逻辑
- 以后换模型非常方便

## 15. 这个项目现在最重要的设计思想

我觉得这个项目最核心的思想有四个：

### 15.1 一切都围绕用户数据组织

知识库、记忆、对话、指标都是按用户隔离的。

### 15.2 不把所有东西都塞进上下文

它通过：

- 检索
- 记忆
- 摘要

来控制上下文大小。

### 15.3 数据不是只存，还要能分析

所以有了 interaction metrics 和 insights 面板。

### 15.4 前端不是简单聊天页，而是一个工作台

菜单、设置、知识库、记忆、指标都在同一个壳里，说明它的目标是做成可持续使用的产品，而不是一次性 demo。

---

## 16. 小结

如果用一句话概括整个项目：

**它是一个以用户知识库和长期记忆为核心，通过 RAG、摘要和指标面板来支撑的多模型聊天工作台。**

你可以把它理解为：

- 聊天负责交互
- 知识库负责外部事实
- 记忆负责长期偏好
- 摘要负责压缩上下文
- 指标负责观察系统效果

这五个部分合起来，才是这套系统真正的完整形态。
